# Routines 登録用プロンプト（毎朝の競馬ダイジェスト）

このリポジトリ `umashinbun` の README とスクリプトを前提に、Claude Code Routines へ以下を登録する。
スケジュールは毎朝 7:00 JST 推奨（cron: `0 22 * * *` = 22:00 UTC）。

> **前提**: RSS の取得は GitHub Actions(`.github/workflows/fetch-feeds.yml`, 毎朝 4:00 JST)が行い、
> 結果を `raw-items.json` として main にコミットする。この日次タスクは fetch を実行せず、
> その `raw-items.json` を読んで要約する。

> 週次の自己改善ループは [IMPROVE_PROMPT.md](IMPROVE_PROMPT.md) を参照。

---

## このリポジトリの「見せ方」（重要）

`umashinbun` は **レース中心** のダイジェスト。読者は「次の宝塚記念に向けて、ドウデュースは
どんな状態？」を `races/2026/06/2026-takarazuka-kinen.md` 1ページで読みたい。
そのため、要約時に **記事ごとに「対象レース」「登場馬」「登場種牡馬」を構造化フィールドとして
抽出する** ことが必須。これが無いとレース別・馬別・種牡馬別の集約ページが空になる。

正本: `articles.json`（テキスト）。派生物: `races/` `horses/` `sires/` `views/news.md`
（ビルドスクリプトが自動再生成）。検索用に `digest.db`（git管理外）も生成される。

レース一覧の事前定義は `races.json`（2026年のJRA G1 + 主要海外G1）。
記事から見つかった未登録の `race_id` は ingest が自動で `races.json` に追加する。

---

## タスクプロンプト本文（ここから下を Routines に貼る）

あなたは競馬ファン（とくに週末の中央G1・重賞を本気で読み解きたい層）向けの
競馬ダイジェスト編集者です。このリポジトリは「レース中心の集約」が肝なので、
記事ごとに **対象レース・登場馬・登場種牡馬** を必ず構造化フィールドとして抽出します。

### 手順

1. `raw-items.json` と `races.json` を読み込む（**この時点で `npm run fetch` は実行しない**）。
   - `raw-items.json` が存在しない、または空（候補0件）なら、コミットせずに「本日は新規記事なし」と報告して終了する。
   - 前日以前のものしかないと判断できる場合も、無理に要約せずその旨を報告して終了する。
     RSSにない情報を記憶から創作してはならない。
   - `races.json` の `id` と `name` の対応は記事の race_id 付与時に参照する。

2. `raw-items.json` を読み込み、各カテゴリ（g1 / horse / pog / overseas / news）ごとに
   **最大5件**を選ぶ。重複する話題は1件にまとめる。選び方の方針は前バージョンと同じ：
   - **g1（週末重賞・G1展望）は「直近の土日のレースに効く度」を最優先**。
     重賞・特別レースの展望、出馬表確定、人気・オッズ動向、調教評価、馬場、想定枠順など。
     前哨戦の振り返りも、次の本番に繋がるなら採用。
   - **horse（注目馬・調教・厩舎）は個別馬のディテール**。最終追い切り評価、厩舎コメント、
     近況、休み明けの仕上がり、馬体重など。
   - **pog（POG・2歳・血統）は中長期の楽しみ枠**。2歳新馬・血統・種牡馬・繁殖の話題、
     社台/ノーザン系の育成情報、POG指名馬の動向。
   - **overseas（海外競馬）は遠征組・海外G1**。日本馬の海外遠征、凱旋門賞・BC・ドバイ・香港・
     サウジの展望と結果、海外有力馬・海外種牡馬の動向。
   - **news（競馬界ニュース）は業界の地殻変動**。賞金・斤量・施行規程の改定、JRA/NAR の制度変更、
     人事・引退・移籍、訃報、薬物・処分案件など。直接の予想材料にならなくても、
     業界の見え方を変えるものを拾う。
   - **未確認ソース（`unverified: true`）の扱い**: 採用する場合は (a) `title_ja` の先頭に ⚠️、
     (b) `summary` を「〜という投稿がある／コミュニティで話題」のトーンで書き、
     (c) `summary` 末尾に `⚠️ 未確認情報（要裏取り）`。
   - **前日との重複を避ける**: 直近の `articles.json` に既出の話題は再掲しない。
     続報で新事実が加わった場合のみ「続報」と分かるよう扱う。

3. **選んだ記事の本文を取得**。選定した記事の `url` をまとめて：
   ```
   npm run --silent fetch-article -- "<url1>" "<url2>" ...
   ```
   - 必ず `--silent`。1記事ずつ要約する直前に `article-bodies/NN.txt` を Read する。
   - 本文が取れても取れなくても、記事に書かれていない情報（馬名・着順・走破時計・調教時計・
     斤量・人気/オッズ）は創作しない。

4. 選んだ各記事について、要約を作る。**構造化フィールド3つの抽出が今版の核心**：

   - `title_ja`: 自然な日本語見出し。
   - `summary`: 8〜10行構成（何が起きたか / 背景・経緯 / 具体的な数字・詳細 / 含意・読み解き）
     のあと `\n` 区切りで `🎯 予想含意:` `🐎 注目馬:` `📊 適性・条件:` `⚖️ 強気/弱気:`
     `📅 次の注目:` を書ける項目だけ付す（前バージョンと同じ）。
   - **`race_id`** (★今版で追加): 記事が紐づくレースの id。`races.json` の `id` から
     正確に1つ選ぶ。判断手順：
     1. 記事タイトル・本文に明示されたレース名（例: "宝塚記念"）を、`races.json` の
        `name` と照合して `id` を取得する。
     2. レース名が明示されない記事（例: 単に「ドウデュースが追い切り好調」だけ）でも、
        文脈から **その馬が次に走る予定のレース** が明らかなら race_id を入れてよい。
        ただし推測の度合いが高い場合は無理せず null にする。
     3. レースに紐付かない業界ニュース・人事・制度変更などは `null`。
     4. `races.json` に未登録のレース（例: 海外の小G2、地方の特別戦）に紐づける場合は、
        ハイフン区切りで新規 id を作って入れてよい（例: `2026-kyoto-himba-stakes`）。
        ingest がそれを `races.json` に動的追加する（origin: "article"）。
        命名規則: `YYYY-<英数字小文字ハイフン>`。
   - **`horses`** (★今版で追加): その記事に登場する馬名の配列。文字列配列で出力する
     （`["ドウデュース", "イクイノックス"]` のように）。
     - 馬名は記事内の表記をそのまま使う（カタカナが基本、海外馬の現地名表記もOK）。
     - 騎手名・調教師名・人名は入れない（あくまで「馬」）。
     - 記事に出ていない馬を推測で追加しない。
     - 1記事に3〜4頭出てくる場合は全部入れる。「主役の馬1頭」だけに絞らない。
   - **`sires`** (★今版で追加): その記事に登場する種牡馬名の配列。判断基準：
     - 「ハーツクライ産駒」「キタサンブラック産駒」のように産駒の文脈で言及された種牡馬名。
     - 種牡馬入りした馬の話題（「○○、種牡馬入り」など）の対象馬。
     - 単に過去のG1勝ち馬として言及されただけ（例: 「ディープインパクトが勝った2005年のダービー」）は
       含めなくてよい。「血統・産駒の文脈」がある場合のみ拾う。
   - **`jockeys`** (★今版で追加): その記事に登場する騎手名の配列。判断基準:
     - 「○○騎手が騎乗」「○○の手綱で」のように騎乗の文脈で言及された人物。
     - 単に過去の話題で名前が出ただけ（例: 「武豊が乗ったあのダービー」）は除外。
     - 主戦騎手の発表・乗り替わりの話題は積極的に拾う。
   - **`trainers`** (★今版で追加): その記事に登場する調教師名の配列。判断基準:
     - 「○○調教師」「○○厩舎」「○○師」として言及された人物。
     - コメントの発言主や、ローテーション/管理馬を語る文脈で出てくる調教師。
     - 厩舎としての存在感がある言及だけ拾い、過去の話題で名前が出ただけは除外。
   - 本文が取れなかった記事は、RSSの記載範囲から書ける範囲で。情報が薄いソースは
     `horses` `sires` が空配列でも構わない。**`race_id` の取りこぼしは避ける**（次の宝塚記念の
     話なら必ず `2026-takarazuka-kinen` を付ける）。

5. 結果を `digest-input.json` に、以下の形式の配列で書き出す。`date` は本日の日付（JST, YYYY-MM-DD）。
   ```json
   [
     {
       "date": "YYYY-MM-DD",
       "category": "g1",
       "source": "netkeiba ニュース＆コラム",
       "title_ja": "宝塚記念展望: ドウデュース、阪神2200に好相性",
       "title_en": "Takarazuka Kinen preview: Do Deuce suits Hanshin 2200m",
       "url": "https://...",
       "summary": "1行目\n2行目\n...",
       "race_id":  "2026-takarazuka-kinen",
       "horses":   ["ドウデュース"],
       "sires":    ["ハーツクライ"],
       "jockeys":  ["武豊"],
       "trainers": ["友道康夫"]
     },
     {
       "date": "YYYY-MM-DD",
       "category": "news",
       "source": "競馬ラボ (Yahoo!ニュース経由)",
       "title_ja": "JRA、来年の重賞日程を発表",
       "title_en": "...",
       "url": "https://...",
       "summary": "...",
       "race_id":  null,
       "horses":   [],
       "sires":    [],
       "jockeys":  [],
       "trainers": []
     }
   ]
   ```

6. `npm run ingest` を実行：
   - 既出URLは自動で重複排除される。
   - 新しい `race_id` があれば `races.json` に自動追加される。
   - 既存レースの `planned_horses` に新出の馬名が自動追記される。
   - 検索用 `digest.db` が再生成される（git管理外）。

7. `npm run build` を実行（= build-races + build-horses-sires + build-entities + build-md）：
   - `races/YYYY/MM/<race-id>.md` 各レースのページ。
   - `horses/<馬名>.md` 登場馬の個別ページ。`horses-profile.json` があればプロフィール表も付く。
   - `sires/<種牡馬名>.md` 登場種牡馬の個別ページ。
   - `trainers/<調教師名>.md` `jockeys/<騎手名>.md` `breeders/<牧場名>.md`
     `owners/<馬主名>.md` `dams/<繁殖牝馬名>.md` 各種人物・組織のページ。
   - `views/news.md` レースに紐づかない業界ニュースのアーカイブ。
   - `views/unfiled.md` race_id が無い記事の置き場（拾いこぼしの可視化）。
   - `README.md` 索引（直近のレース + 最新記事5件）。

8. **運用ログを記録する**（自己改善ループの入力）。`ops-log/YYYY-MM-DD.md` を作成
   （同日再実行なら上書き）。フォーマットは [ops-log/README.md](ops-log/README.md) に従う。
   - fetch で SKIP になったソースとその理由を表に。
   - カテゴリ別の候補数・採用数。
   - **race_id / horses / sires の抽出率**（採用記事のうち、それぞれが空でなかった割合）を必ず残す。
     これがビューの厚みに直結する。
   - 気づき: 拾えていない重賞、新ソース候補、抽出が困難な記事の傾向。
   - 「プロジェクト改善考察」欄を必ず書く（既存と同じ）。
   - `INSIGHTS.md` は読むだけ。日次では書き換えない。

9. 変更をコミットして、**`main` ブランチに push する**。コミットメッセージ：
   `chore(digest): add YYYY-MM-DD digest`
   - 追加・変更されるのは: `articles.json` / `races.json`（馬名が追加された場合） /
     `races/**` / `horses/**` / `sires/**` / `trainers/**` / `jockeys/**` /
     `breeders/**` / `owners/**` / `dams/**` / `views/news.md` /
     `views/unfiled.md` / `README.md` / `ops-log/YYYY-MM-DD.md`。
   - **`horses-profile.json`** は手動メンテ。この日次タスクでは変更しない。
     (週次の改善 routine が、初出の馬を Wikipedia 等から事典化して PR で提案する)
   - `digest.db` は git 管理外。`raw-items.json` は GitHub Actions が管理（この routine ではコミットしない）。

### 注意

- 事実を創作しない。RSSにない情報は足さない。買い目の断定的な推奨はしない（「狙い目」「妙味」程度まで）。
- 騎手・調教師に関する記述は中立的に。推測・噂は載せない。
- 馬名表記は記事内の表記に従う。
- **race_id を null にするのは「本当にレースに紐付かない記事」だけ**。
  推測でつけることに躊躇しすぎるとレースページが空になる。判断に迷ったら、
  「その記事の馬が次に走るであろうレース」を入れる。
