# ops-log — 運用ログ

毎日の日次ダイジェスト routine が、その日の収集・要約状況を `YYYY-MM-DD.md` として
ここに書き出す。週次の改善 routine ([../IMPROVE_PROMPT.md](../IMPROVE_PROMPT.md)) が
このログを集計して `src/feeds.ts` や `ROUTINES_PROMPT.md` の改善 PR を作る、
という自己改善ループの記録場所。

人間がコードを直す必要はない。ログは routine が自動で残す。

## フォーマット (日次 routine が生成)

```markdown
# 運用ログ YYYY-MM-DD

## モード判定
- mode: race_week (例)
- target_races: [ "宝塚記念 (2026-06-14, 7日後, 17頭)" ]
- next_month_races: []
- notes: [...]

## fetch 結果
| ソース | カテゴリ | 結果 | 取得/新規 | 備考 |
|---|---|---|---|---|
| netkeiba ニュース＆コラム | g1 | OK | 30 / 12 | |
| 東スポ競馬 (Yahoo!ニュース経由) | g1 | OK | 50 / 8 | |
| 競馬ラボ (Yahoo!ニュース経由) | g1 | SKIP | - | HTTP 403 |
| ... | | | | |

- 成功ソース数: X / 全Y
- 候補総数: N件

## カテゴリ別 採用状況
| カテゴリ | 候補 | 採用 | 備考 |
|---|---|---|---|
| g1 | 30 | 4 | 宝塚記念関連を優先 |
| horse | 25 | 3 | 出走予定馬の追い切り中心 |
| pog | 8 | 1 | 新馬戦の話題 |
| overseas | 40 | 2 | BC Classic 関連 |
| news | 5 | 0 | 採用なし |

## 構造化抽出率 (重賞週/月末モードの肝)
| フィールド | 採用記事中の非空率 | 備考 |
|---|---|---|
| race_id | 8 / 10 (80%) | |
| horses | 9 / 10 (90%) | |
| sires | 4 / 10 (40%) | sires は記事に依存 |
| jockeys | 6 / 10 (60%) | |
| trainers | 5 / 10 (50%) | |

## 事典化した馬 (race_week / monthly_prep モードのみ)
- horses-profile.json に追加: 2頭
  - シェイクユアハート (Wikipedia URL)
  - タガノデュード (Wikipedia URL)
- 取得失敗: 1頭
  - ファミリータイム (Wikipedia 項目なし)

## 出走予定馬 / 結果取得の成否 (race_week モードのみ)
- B-0 planned_horses 更新: 宝塚記念 17頭 → 18頭 (シックスペンス追加)
- B-0-2 結果取得: 該当なし (今日終わったG1は無し)
- WebFetch 失敗: なし

## 気づき / 改善候補
- 競馬ラボが継続して 403。サイト改編の可能性、 代替フィード要検討。
- pog カテゴリは件数が薄い。 2歳新馬戦の専用ソース追加を検討。
- (新ソース候補があればURLとともにメモ)

## プロジェクト改善考察 (収集メモではなく「仕組みをどう良くするか」の所感)

その日の収集結果を踏まえ、**プロジェクトそのものの改善**について考えたことを書く。
ニュース内容の感想ではない。以下の問いを毎日たたき台にする (全部埋めなくてよい、
その日に言えることだけ)。過去のログ (特に `INSIGHTS.md`) を踏まえ、 既出の論点は
「継続」「悪化」「解消」のように経過が分かるよう書く。

- **fetch の仕組みは妥当か**: 取得・鮮度フィルタ (36h)・重複排除は意図通り効いているか。
  取りこぼし / 取りすぎ / 弾きすぎの兆候はないか。
- **本当に重賞週に厚い情報が集まっているか**: race_week モードで対象レース関連の記事が
  ホワイトリスト馬中心に拾えているか。 過去回顧の馬まで `horses` に混入していないか。
- **ソース構成の穴**: 国内/海外、 専門紙/総合紙の観点で欠けている視点はないか。
  逆に重複・低品質で枠を食っているソースはないか。
- **構造化抽出の精度**: race_id / horses / sires / jockeys / trainers の非空率が
  期待通りか。 特に race_id が null になりやすい記事の傾向はあるか。
- **馬の事典化が機能しているか**: 新出馬の Wikipedia 取得は成功しているか。
  Wikipedia に項目が無い馬の扱いは妥当か。
- **planned_horses / results の最新化**: 重賞週モードの B-0 / B-0-2 が動いているか、
  失敗パターンはあるか。
- **パイプライン全体**: routine の手順・出力形式・運用に、無駄や壊れやすい点はないか。

> ⚠️ ここは「記録」に徹する。 コード (`src/feeds.ts` 等) や `ROUTINES_PROMPT.md` は
> この日次タスクでは変更しない。 実際の改善 PR は週次 routine
> ([../IMPROVE_PROMPT.md](../IMPROVE_PROMPT.md)) が行う。
```

## INSIGHTS.md (累積考察・追記式)

- `ops-log/INSIGHTS.md` は、週次の改善 routine が日次ログの「プロジェクト改善考察」を
  集約して **上書きせず追記していく** 累積ファイル。日次ログは1日で流れるが、
  こちらには「経時的に見えてきたこと・繰り返す問題・検証したい仮説・打った手とその後」が残る。
- 日次 routine は INSIGHTS.md を **読んで** 考察の土台にしてよいが、
  **書き込むのは週次 routine だけ** (日次が追記すると競合・肥大化するため)。

## DEBATES/ (週次の対立議論ログ)

- `ops-log/DEBATES/YYYY-MM-DD.md` は、週次改善 routine が
  「ブレスト → 推進派⇄懐疑派 → ジャッジ」 で改善案を絞り込んだ過程の **逐語議事録**。
- 日次 routine はこれも読むだけでよい。 書き込むのは週次 routine。

## 運用ルール

- 1日1ファイル。同日再実行時は追記ではなく上書き更新でよい。
- このログ自体はコミット対象 (改善 routine が読むため)。
- 日次 routine が `INSIGHTS.md` / `DEBATES/` に書き込むことは禁止 (読むだけ)。
