## 気づき / 改善候補

1. **記事本文フェッチが全滅 (403)**: `fetch-article.ts` のUser-Agentでは全主要日本語競馬サイト・Yahoo!ニュース・BloodHorse・TDNがブロック。RSS description（数百字）ベースで要約を作成する運用に事実上なっている。本文を取得できれば要約の精度は大幅に向上するが、現状の手段では困難。
2. **Wikipedia も403**: `ja.wikipedia.org` への WebFetch が全て403。馬の事典化はWebSearchサマリーに依存している。Wikidataの API（`https://www.wikidata.org/w/api.php`）は別途試す価値あり。
3. **宝塚記念出走予定馬の更新失敗**: JRA公式の horses.html が403。出馬表確定（6/11）後に次回実行で更新を再試行すること。
4. **2着・3着のタイム取得が困難**: WebSearchの結果には1着タイムが含まれることが多いが、2着以下は馬身差の記述にとどまる。タイムのnullフィールドが増えるのは構造化データとして問題ではないが、今後の対策として Wikipedia や netkeiba データベースページからの取得を検討。
5. **horse カテゴリの採用率が低い (1/58)**: 多くが安田記念の重複記事（同じ情報を複数ソースが報道）。重複度の高いカテゴリ内ソースを絞ることも検討。

## プロジェクト改善考察

INSIGHTS.md の論点トラッカーを参照しながら、初めての実稼働で見えた問題を整理する。

**fetch の仕組み**（🟡継続 → 🔴悪化）  
記事本文フェッチ (fetch-article.ts) が全件403となった。GitHub Actions の RSS 取得は動いているが、記事本文の取得がこのサーバー環境では機能しない。RSS description だけで要約を構築する限界がある。User-Agent を変えても効果は不明。Cloudflare系のサイトは特にブロックが厳しい。→ 週次改善で「本文取得の代替手段」を検討する価値がある（Google Cache、archive.org など）。

**重賞週に厚い情報が集まっているか**（概ね良好）  
安田記念当日という最高のコンテキストで、RSS には多くの安田記念関連記事が含まれており選定は容易だった。ただし本文が取れないため「表彰式」「コメント」程度の薄い要約になっている。RSS に含まれる description だけでも事実確認は可能だが、追い切り記事（前走コメントや調教内容）の詳細は取れない。

**ソース構成の穴**（🟡継続）  
overseas カテゴリは82件と多いが、英語記事の要約は日本語競馬ファン向けとしては深くない。TDNは日本レースを事前分析しており有用。BloodHorse は結果速報のみで詳細分析が少ない。日本語で競馬を語るユーザーにとっては、Yahoo!ニュース（東スポ・スポ報知・netkeiba）が最も価値があるが、そこが全件403という現状。

**構造化抽出の精度**（良好 62〜92%）  
race_id の非空率62%は、海外記事・pog記事にはrace_idが付かないことを考えると妥当。日本重賞記事では race_id は全件付与できている。horses の92%は高い。過去回顧の馬を意図的に除外できているかは今後確認が必要。

**馬の事典化**（🔴悪化）  
Wikipedia へのアクセスが全てブロック。WebSearch のサマリーでプロフィールを構築したが、詳細な重賞成績（2着・3着の漏れ）や通算成績の正確性が低い。特に2着・3着馬（パントルナイーフ等）はほぼプロフィールゼロの状態。→ 次週の改善 routine で Wikipedia API またはモバイルエンドポイントへの切り替えを提案する。

**planned_horses / results の最新化**（🔴悪化→一部解消）  
B-0-2 (結果取得) は WebSearch を使うことで4レース分の結果を入力できた。ただし B-0 (planned_horses の JRA 公式サイトからの取得) は403でブロックされた。宝塚記念の最新出走予定馬は次回（6/8以降）に再試行が必要。

**パイプライン全体**  
今回が実質的な初回実稼働。設計は概ね機能しているが「記事本文が取れない」「Wikipedia が取れない」という根本的な制約が露顕した。RSS description ベースの運用でも一定の品質は出せるが、詳細な追い切りコメントや騎手コメントの本文が取れないのは重賞週モードの価値を下げる。→ INSIGHTS.md の「fetch の仕組み」論点を悪化として更新（週次 routine で追記）。
