### 論点 2: BloodHorse pubDate=undefined に fetchedAt フォールバックを追加する

**🔵 推進派 T1**:  
7日連続・55件/日規模で BloodHorse の pubDate が undefined のまま raw-items.json に出力されている。candidates.push() 時に `pubDate: item.pubDate || new Date().toISOString()` のようなフォールバックを入れれば、少なくとも取得日時が pubDate として記録される。routine 側での日付表示・ソートに使えるようになり、「日付不明」状態を解消できる副次効果がある。rss.ts 修正の前段階として実施価値がある。

**🔴 懐疑派 T2**:  
`isFresh()` は `candidates.push()` より **前** に呼ばれている（fetch-feeds.ts 84行目: `if (!isFresh(item.pubDate, nowMs)) { stale++; continue; }` → 89行目: `candidates.push(...)` の順）。したがって candidates.push() 時に fetchedAt を埋めても、freshness フィルタの動作には一切影響しない。BloodHorse の pubDate=undefined の根本原因は、実際の XML で使われている日付タグ（`<pubDate>` 以外の可能性）に rss.ts の tagContent が対応していないことと推察される。原因を特定せずに fetchedAt を当て貼りしても、rss.ts が正しく読めれば自動解決する問題を永続化させるだけ。isFresh() への影響なし・根本解決でもない・half-finished になるリスクが高い。

**🔵 推進派 T1**:  
懐疑派の指摘（isFresh() は push より前に呼ばれる）は正確。freshness フィルタへの影響がないことは認める。ただし raw-items.json に fetchedAt が記録されることの副次効果は実在する。BloodHorse の XML フォーマット確認はクラウド環境から 403 で実行不能なため、現時点では rss.ts 修正の方針が立てられない。その間の暫定措置として fetchedAt フォールバックを入れる価値はある。

**🔴 懐疑派 T2**:  
「暫定措置」という位置づけが問題。raw-items.json の pubDate 欄に fetchedAt が入るようになっても、routine 側がそれを活用する設計変更が別途必要になる。今実装しても half-finished になる可能性が高い。IMPROVE_PROMPT.md の「必要最低限のコード変更に留める」原則に照らしても、副次効果のみの変更は正当化しにくい。BloodHorse の XML を GitHub Actions 環境またはローカルで curl 確認し、rss.ts に正しいタグ対応を追加するのが正道。今週はスキップして 🔵検証中 を継続すべき。

**⚖️ ジャッジ裁定**:  
結論: **やらない**  
懐疑派の論点（isFresh() は push より前に呼ばれる・根本原因は rss.ts のタグ未対応・fetchedAt は half-finished）がいずれも正確で、推進派の反論（副次効果・暫定改善）より説得力が高い。今回は実施しない。BloodHorse XML フォーマットは GitHub Actions 環境またはローカルで curl 確認してから rss.ts を修正するのが正道。論点トラッカーの状態は 🔵検証中 を継続（ローカル curl 確認待ち）。

**🔵 推進派 T1**:

**⚖️ ジャッジ裁定**:
