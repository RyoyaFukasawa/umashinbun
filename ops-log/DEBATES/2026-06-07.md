### 論点 1: 競馬ラボ (Yahoo!ニュース経由) を feeds.ts から完全削除する

#### 🔵 推進派 — T1
GHA の raw-items.json でゼロ件、HTTP 403 の完全死亡が確認済み。死んだエンドポイントを feeds.ts に残すことは技術的負債。毎回のフィード取得処理で403 レスポンスの待機・エラーハンドリングがパイプライン全体の実行時間を引き延ばし、エラーログに403が混入し続けることで本当に異常なエラーが埋もれる。g1 カテゴリは netkeiba と東スポの2本が生きており、削除後もカテゴリ空洞化は起きない。先送りこそがリスク。

#### 🔴 懐疑派 — T1
HTTP 403で0件取得でも実害はゼロ（システムは落ちていない）。Yahoo! RSSは一時的なアクセス制限や認証変更で403になるケースがあり、復活の可能性を完全に捨てることになる。feeds.ts を変更すれば GHA パイプライン再実行コストと人的ミスのリスクが生まれる。コメントアウトで様子見が最善（削除は最後の手段）。

#### 🔵 推進派 — T2
「2本体制で耐障害性が下がる」という懐疑派の論は錯覚（現在も実質1本体制）。HTTP 403はアクセス拒否であり復活根拠なし。「コメントアウト vs 完全削除」について：コメントアウトは grep・補完候補に毎回現れ続け認知コストを永続的に払い続ける。削除して Git 履歴に残す方が明快。

#### 🔴 懐疑派 — T2
「Gitで復元できる」は正しいが、コミットメッセージもコメントと同じ認知コストを別の場所に移しているだけ。削除は再実装コストが非対称に高い（コメントアウトの方が簡単に戻せる）。JRAアクセス制限は突然解除される実績あり（スクレイピング界隈では数ヶ月後に復活が珍しくない）。「非対称なリスクを軽視するな」。

#### 🔵 推進派 — T3
「削除は不可逆」は Git を使っている限り成立しない。`git log`・`git show`・`git checkout` で任意のコミットから復元できる。Gitの履歴は日常的なコードリーディングに割り込まない。コメントアウトされたコードは毎回のgrep・補完候補に現れ続ける。「動かないコードはコードではなく負債である」。

#### ⚖️ ジャッジ裁定 — 結論：**やる（完全削除）**

**決め手：推進派T2・T3**

懐疑派T2の「JRAアクセス制限が突然解除される実績あり」は競馬ラボ（Yahoo!ニュース経由）に対して根拠が薄い（JRA公式とは別物）。懐疑派T1の「実害ゼロ」は正確だが「実害がないから負債を抱えてよい」は技術的判断として成立しない。

「コメントアウト vs 完全削除」：完全削除が正しい。推進派T3の指摘が決定的——コメントアウトされたコードは grep や補完候補に出続ける。Git の履歴で復元できる以上、懐疑派T2の「削除＝不可逆」という前提は崩れており、再実装コストの非対称性の論拠も成立しない。

**条件**：削除時のコミットメッセージに「403確認日・競馬ラボURL・復活時の参照先候補」を明記することで懐疑派の懸念を吸収。

### 論点 3: ROUTINES_PROMPT.md の要約フォーマットに overseas 英語記事の日本語化ルールを追記する

#### 🔵 推進派 — T1
現状、overseas 英語記事は「日本語化困難」として採用が抑制されており、83件/日という豊富な海外情報源が活用されていない。問題の本質はAIの能力ではなくプロンプトに明示的な指示がないこと。「英語記事でも日本語要約・カタカナ馬名義務化」を明記すれば、判断コストがゼロになり採用率が向上する。追記はわずか1段落、コストは最小・効果は最大。

#### 🔴 懐疑派 — T1

#### 🔵 推進派 — T2
「動いているから不要」は危険な論理。暗黙の挙動はモデルバージョン更新や担当者交代で消滅するリスクがある。文書化は「現状の固定」ではなく「意図の宣言」。プロンプト肥大化の懸念は構造化で対処可能で干渉リスクは設計の問題。予防的明示化のコストは軽微、リスクは非対称。

#### ⚖️ ジャッジ裁定 — 結論：**やる（制約付き実装）**

**決め手：推進派T2**

暗黙の挙動は「現在たまたま動いている」に過ぎず、モデル更新・プロンプト変更のいずれかで無音消失するリスクがある。「課題駆動で対処」は障害発生後の検知が困難なケースに不向き。
