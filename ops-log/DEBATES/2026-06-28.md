**背景**: npm run fetch を実行し raw-items.json を分析した結果、BloodHorse 3フィードから取得した全55件が pubDate=undefined だった（100%）。現在の isFresh() は undefined を true として扱う（意図的設計）。問題は pubDate が undefined であることで、鮮度が不明なまま記事が通過していること。

---

**🔵 推進派 T1**:

主張A（問題の重大性）: 全55件 100% で pubDate が取得できていない。これは偶発的な問題ではなく、BloodHorse の RSS XML 構造と rss.ts のパーサーが一致していないことを示す構造的問題。`isFresh()` が undefined を通す設計はフォールバックとして機能しているが、「今日の記事」と「2週間前の記事」が区別できないまま候補に入る問題は品質上の欠点。

主張B（修正方針の候補）: BloodHorse が RSS 2.0 ならば `<channel>/<lastBuildDate>` がフォールバック候補。Atom ならば `<feed>/<updated>` が候補。`<dc:date>` のネームスペース解決に失敗している可能性もある。rss.ts の正規表現パターンを確認すれば原因が特定できる。

主張C（実装難易度）: rss.ts は軽量なパーサーで200行程度。修正箇所はアイテムパースの日付取得ロジックのみ。影響範囲は限定的。

---

**🔴 懐疑派 T1**:

主張A（XML フォーマット未確認）: BloodHorse の RSS XML を実際に curl で取得して確認しない限り、「何が正しい修正か」が特定できない。`<pubDate>` タグがアイテムレベルに存在しない可能性、タグ名が異なる可能性（`<dc:date>` の namespace 不一致等）、Atom フォーマットで `<published>` が存在する可能性——これらを区別せずに「rss.ts を修正する」と言っても、何を修正するかが決まらない。

主張B（現在の設計は意図的）: `isFresh()` が undefined を true として返す設計は、「日付不明で良い記事を落とすより、routine の選定に委ねる」という意図的な判断（fetch-feeds.ts のコメントにも明記）。この設計を変更することはシステムの哲学を変えることでもある。

主張C（このクラウド環境では検証不可）: npm run fetch が全件 HTTP 403 で失敗する環境では、修正後の動作確認ができない。「未検証の修正」をマージするリスクが高い。

---

**🔵 推進派 T2**:

反A（XML 確認を「保留解除条件」にする提案）: 推進派は「今すぐ修正する」を主張するのではなく、「BloodHorse XML を curl で確認することを次のアクションとする」として保留解除条件を定義する。条件: ① BloodHorse RSS をローカル環境または curl で取得し、item レベルの日付タグ有無を確認 ② 確認結果に基づき rss.ts の修正方針を決定。

反B（isFresh の設計変更ではない）: 提案は「isFresh() の動作を変える」のではなく、「pubDate が正しく取得できるようにする」こと。undefined のまま通すかどうかは別の判断。

---

**🔴 懐疑派 T2**:

反A（推進派自身が「保留が妥当」を認めた）: 推進派 T2 が「保留解除条件を定義する」と述べた時点で、現時点での実施は適切ではないことを認めている。これは「やる」ではなく「条件付き保留」の主張と等価。

---

**⚖️ ジャッジ裁定（論点2）**:

結論: **保留**

決め手: **XML フォーマット未確認の状態での修正は「何を修正するか」が決まらない（懐疑派 T1 主張A）**。推進派 T2 の「保留解除条件を定義する」への移行がその証左。

**保留解除条件**:
1. BloodHorse の RSS フィード（例: `https://www.bloodhorse.com/feeds/rss/allnews`）を curl または Wget で取得し、`<item>` タグ内の日付タグの有無と形式を確認する
2. 確認結果を次回週次改善議事録に記録し、rss.ts の修正方針（フォールバックタグの追加 or ネームスペース修正 or 別途日付フィールド追加）を決定する
3. このクラウド環境（GH Actions または curl が通る環境）で修正後の動作確認ができること
