# 週次改善プロンプト（umashinbun の自己改善）

このリポジトリ `umashinbun` を、過去1週間の運用ログ（`ops-log/`）をもとに改善するための
週次 Routine 用プロンプト。**直近7日の日次ログを全件読んだ上で**、(A) 複数レンズのagentで
**ブレスト（発散）** → (B) 推進派⇄懐疑派の **対立議論＋第3のジャッジ裁定（収束）** を行い、
その結論に基づくコード変更を **PR として提案** する（ブレスト/議論ログもPRに添える）。
main への直接コミットは `ops-log/**` の記録のみ。コード変更は必ず PR にする。

スケジュール: 毎週月曜 8:00 JST 推奨（cron: `0 23 * * 0` = 日曜 23:00 UTC）。

---

## タスクプロンプト本文（ここから下を Routines に貼る）

あなたは umashinbun（毎朝、週末の競馬・注目馬・調教・血統・海外競馬・業界ニュースを
翻訳・要約して蓄積するプロジェクト）の運用改善担当です。過去1週間の運用ログを分析し、
収集の質を上げる改善を **PR として提案**してください。
Node 22.5以上が必要で外部依存はありません（npm install 不要）。

### 手順

1. **直近7日分の日次ログを全件読む**。`ops-log/YYYY-MM-DD.md` を1件ずつ Read し、各ログの
   「fetch結果」「本文取得成否」「気づき/改善候補」「プロジェクト改善考察」まで目を通す。
   累積考察 `ops-log/INSIGHTS.md` も読む。
   - **何件・どの日付を読んだかを記録する**（後段のブレスト/議論ログ冒頭に全件リストアップする）。
   - ログが1件も無ければ「分析対象のログなし」と報告して終了する（PRは作らない）。
   - これらの日次ログ＝現場の一次データが、このあとのブレスト（発想）と対立議論（裁定）の根拠になる。
     週次の判断は記憶や一般論ではなく、必ずこの7日分のログに根ざすこと。

2. ログを集計し、以下の観点で問題と改善余地を洗い出す。
   - **恒常的に失敗するフィード**: 直近で繰り返し SKIP（HTTP 403/404/タイムアウト等）になっているソース。
     URLのサイト改編が疑われるものは、現行URLの代替（公式の現行RSS URL）を Web で調べて確認する。
   - **カテゴリの偏り・候補枯渇**: 特定カテゴリで採用件数が継続的に少ない（候補が薄い）。
   - **重複・低品質**: 同じ話題ばかり、要約しづらい質の低いソースが多い。
   - **日次の「プロジェクト改善考察」欄**を読み、繰り返し出ている論点・経時的な変化を拾う
     （単発の気づきと、何日も続く構造的な問題を区別する）。

2.5. **累積考察 `ops-log/INSIGHTS.md` を更新する**（このファイルはコミット可。PRとは別に main へ残す）。
   - **論点トラッカー**の表を更新する: 既出論点の状態を `🟢解消 / 🟡継続 / 🔴悪化 / 🔵検証中` で遷移させ、
     `最終更新` 日を直す。今週新たに見えた構造的論点は行を**追加**する。**既存行は消さない**（経過を残す）。
   - **週次エントリ**を末尾に**追記**する（`### YYYY-MM-DD — 見出し`）。その週に考えたこと、
     最新性/先進性は担保できているか（一次情報 vs 報道の後追い）、ソース構成の穴、選定ロジックの妥当性、
     打った手（このあと出すPR）と残した宿題を書く。**過去のエントリは上書きしない**。
   - これは「考察の蓄積」であり、PRを出さない週でも更新してよい（むしろ更新する）。

> **エージェント起動の実行制約**（クラウドのスケジュール実行＝Routines環境の仕様。必ず守る）:
> - **サブエージェントは子エージェントを産めない**（アーキ上の制約）。ブレストの各レンズ・推進派・
>   懐疑派・ジャッジは、**すべてこのメインタスク（オーケストレーター）が Agent ツールで直接立てる**。
>   サブエージェントの中でさらに Agent を呼ぶ入れ子構成にしない（その呼び出しは失敗する）。
> - 並列起動は **同時 〜8 体まで**に抑える（環境の並列上限〜10・1体あたり約20kトークンのオーバーヘッドを考慮）。
>   ブレスト3体・議論3体は1論点ずつ順に回せば十分収まる。
> - サブエージェントには結論だけでなく**発言の逐語**を返させ、メインがそれを DEBATES ログに転記する
>   （議事録はメインが組み立てる。サブ同士は直接会話できないので、往復はメインが仲介して渡す）。
> - 万一サブエージェントが使えない実行環境なら、**1コンテキスト内で役を順に演じて**でも議事録を残す
>   （独立agentより質は落ちるが、空で終わらせない）。
>
> **トークン予算のハード上限**（青天井禁止。週次の無人実行でコストが暴走しないための上限。
> 「もう少し詰めたい」と感じても以下を超えない）:
> - **論点は最大3件**（手順3-Bで選ぶ件数。2件で十分なら2件でよい）。
> - **ブレスト(3-A)は最大3ラウンド**（1ラウンド = 3レンズが1発ずつ発言）。出尽くす前でも3ラウンド到達で打ち切る。
> - **対立議論(3-B)は1論点あたり最大4ラリー**（1ラリー = 推進派↔懐疑派の往復1回）。
> - **早期終了条件**: 以下のいずれかに該当した時点で、上限到達前でも打ち切ってジャッジに渡す：
>   (i) 同じ主張の繰り返しが2回続いた、(ii) 合意に至った、(iii) 平行線が明確になり論点が出尽くした。
> - ジャッジは1論点につき1回のみ呼び出す（再裁定しない）。
> - サブエージェント1体あたりの応答は**逐語議事録部分を含めて目安1500字以内**で要求する
>   （プロンプトに「N字以内で」と明記して節約させる）。

3. **発散→収束で改善を練る**（このプロジェクトの肝。1体で結論を出さず、独立した複数agentに役割を
   割り当てる）。記録先は `ops-log/DEBATES/YYYY-MM-DD.md`（フォーマットは
   [ops-log/DEBATES/README.md](ops-log/DEBATES/README.md)）。**全agentに「直近7日の日次ログ（手順1で
   読んだもの）＋INSIGHTS.md」を材料として渡す**（記憶や一般論でなく現場データに根ざさせる）。

   **3-A. ブレスト（発散・批判禁止・量重視）** — Agent ツールで以下の3つのレンズのサブエージェントを立て、
   **互いの案に乗っかり合いながら**アイデアを広げさせる。良し悪しは判定しない（批判禁止）。
   - **🔭 ソースハンター**: 拾えていない一次情報源・カテゴリ・地域・非英語圏などを提案する。
     「今のソース構成のどこに穴があるか」を日次ログのfetch結果から具体的に挙げる。
   - **🙋 ユーザー代弁（競馬ファン読者の視点）**: 読み手にとっての価値・不足を出す。薄い/重複/役に立たない
     枠の指摘、逆に「週末予想にこれが読みたいのに無い」「個別馬の現場情報が薄い」等を挙げる。
   - **💡 斬新枠（ワイルドカード）**: 既存の枠組み自体を疑う飛んだアイデアを出す。取得頻度の変更、
     新カテゴリ、全く別の角度、思い切った再設計など。突飛でも構わない（収束はフェーズBで行う）。
   - **3レンズを往復させ、互いの案に「それならこういう案も」と乗っかる連鎖を作る**。**最大3ラウンド**
     （1ラウンド = 3レンズが1発ずつ）で打ち切る——出尽くしていなくても3ラウンドを超えない（青天井禁止）。
     **やり取りは要約せず、発言を実際のまま・発言順に全部記録する**（議事録）。各発言に「レンズ絵文字・Tn」
     を付け、「なぜそう思ったか」「誰の案に乗ったか」を省かない。出力を `ops-log/DEBATES/YYYY-MM-DD.md`
     の「フェーズA」節（ブレスト議事録）に書く。

   **3-B. 対立議論＋裁定（収束）** — フェーズAの新規アイデア＋手順2で拾った日次の既存論点を合わせ、
   **重要・継続しているものを最大3件**選ぶ（2件で十分なら2件でよい。欲張らない）。各論点について Agent ツールで3体を立てる：
   - **🔵 推進派（Proposer）**: 「この改善をやるべき」と具体策・根拠・期待効果を主張する。
   - **🔴 懐疑派（Skeptic）**: 「やらない方がいい／優先度が低い」を、リスク・コスト（token含む）・
     反証・副作用から主張し、推進派の具体策の穴を突く。
   - **推進派と懐疑派を往復させる。1論点あたり最大4ラリー**（1ラリー = 推進派↔懐疑派の往復1回）で打ち切る。
     ただし以下のいずれかに該当した時点で上限到達前でも打ち切る：(i) 同じ主張の繰り返しが2回続いた、
     (ii) 合意に至った、(iii) 平行線が明確になり論点が出尽くした。**青天井禁止**——「もう少し詰めたい」と
     感じても4ラリーを超えない。各々が相手の直前の発言を読んで応答する。**やり取りは要約せず、
     発言を実際のまま・発言順に全部記録する**（議事録）。各発言に「🔵/🔴・Tn（ターン番号）」を付ける。
     譲歩や前提修正などの論の動きも省かない。
   - **⚖️ ジャッジ（Judge）**: 逐語の議論ログ全文を渡し、中立に **「やる / やらない / 保留」** を
     裁定させる。**どのターンのどの発言が決め手か**をターン番号で参照させ、保留なら何を観測すれば
     結論が出るかを述べさせる。
   - 議論は**事実ベース**にする。フィードURLの生死・HTTPコード・候補件数などは、agentに渡す前に
     実際に `npm run fetch` 等で確かめ、検証済みの事実を材料として与える（憶測の殴り合いにしない）。
   - ブレスト出力・**逐語の議論ログ（全ラリー）**・ジャッジ裁定をすべて `ops-log/DEBATES/YYYY-MM-DD.md`
     に書き出す（フォーマットは [ops-log/DEBATES/README.md](ops-log/DEBATES/README.md)）。
     **冒頭に「読んだ日次ログ全件（日付リスト）」を記す**。
   - **このログ自体は main に直接コミットしてよい**（`ops-log/**` の領分）。

4. ジャッジが **「やる」** と裁定した改善だけを実装する。典型的には `src/feeds.ts` の `FEEDS` を編集する：
   - 死んでいるフィードの URL を、確認できた現行 URL に **差し替える**。
   - 候補が薄いカテゴリに、信頼できる新しいRSSソースを **追加する**（無料・安定・日次更新を優先）。
   - 機能していないソースを削る。
   - 新ソースを追加・差し替える前に、実際にそのURLを `fetch` して **RSS/Atom として妥当か必ず検証**する
     （HTTP 200 かつ XML が返り `<item>`/`<entry>` を含むこと）。検証できないURLは採用しない。
   - 「やらない／保留」となった論点は**実装しない**。保留は手順2.5で INSIGHTS.md の論点トラッカーに
     `🔵検証中` として残し、「次に何を観測すれば結論が出るか」を書いておく。

5. 変更を加えたら `npm run fetch` を実行し、フィード取得が以前より改善している（失敗が減る/候補が増える）ことを確認する。
   - 生成された `raw-items.json` は中間生成物なのでコミットしない。

6. 改善を **新しいブランチ**で行い、**PR を作成**する。main へ直接 push しない。
   - **先に手順2.5(INSIGHTS.md)と手順3(DEBATESログ)の `ops-log/**` 更新を main にコミット**してから、
     コード改善用ブランチを切る（考察・議論の蓄積とコード提案を分離する。`ops-log/**` は PR ブランチに含めない）。
   - ブランチ名: `improve/feeds-YYYY-MM-DD`
   - PR ブランチで変更してよいのは原則 `src/feeds.ts`（必要なら `scripts/` の収集ロジックも）。
   - **`digest.db` / `digests/**` / `views/**` / `README.md` / `ops-log/**` は PR では変更しない**
     （`ops-log/**` は手順2.5・3で main に直接コミット済み。その他は日次ダイジェスト routine の領分）。
   - コミットメッセージ例: `improve(feeds): replace dead サンスポZBAT! feed, add 競馬ブック`
   - PR 本文には必ず以下を書く：
     - **議論の結論**: 今週ジャッジが「やる」と裁定した論点と、その決め手（1〜2行ずつ）。
       根拠となった議論ログ `ops-log/DEBATES/YYYY-MM-DD.md` への**リンクを必ず貼る**（レビュー時に議論を辿れるように）。
     - **変更内容**: 差し替え/追加/削除したソースと、その現行URLの検証結果（HTTPステータス等）。
     - **before/after**: 変更前後の `npm run fetch` の成功ソース数・候補件数。
   - PR 作成には `gh pr create` を使う。

7. 「やる」と裁定された改善が無い週でも、**手順2.5(INSIGHTS.md)と手順3(DEBATESログ)は行う**
   （考察・議論の蓄積はPRの有無と独立）。それらを main に直接コミットしてよい。
   コード改善が無ければ PR は作らず「今週はコード改善不要。議論ログと考察は ops-log に記録済み」と報告して終了する。

### 注意

- **必ず直近7日の日次ログを全件読んでから**ブレスト・議論する。発想も裁定も現場の一次データ（日次ログ）に
  根ざさせる。記憶や一般論で語らせない。読んだログの日付は DEBATES ログ冒頭に全件記す。
- **ブレストは発散、議論は収束**。ブレスト（3-A）は批判禁止・量重視でアイデアを広げ、対立議論（3-B）で
  絞り込む。両者を混ぜない（ブレスト中に「でもリスクが」と言わせない）。
- **議論は本物の対立にする**。1体に「賛成と反対の両方を書いて」と頼むと中庸に流れる。必ず独立した
  agentに役割を固定して立てる。ジャッジも別agentにし、推進派/懐疑派の発言を読んでから裁定させる。
- **コード変更は必ず PR にする。main に直接コミット・push しない**（ただし `ops-log/**` の記録は main 直コミット可）。
  マージ判断は人間が行う。
- 新ソースは実際に取得検証してから採用する。動作未確認のURLを推測で足さない。
- 1度のPRで盛り込む変更は欲張りすぎない（レビューしやすい粒度に）。
- ソースを足す場合は、信頼できる大手紙・専門紙・公式機関を優先する（個人ブログやSNSは原則 `unverified: true`）。
